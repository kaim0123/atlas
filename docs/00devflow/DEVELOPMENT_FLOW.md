# 開発の流れ

最終更新: 2026-07-03

一般的なアプリ開発は、おおむね以下の工程で進む。各工程の成果物は `docs/` 配下にファイルとして残していく想定。

## 1. 要件定義 (Requirements Definition) — 完了

「何を作るか」「何を作らないか」を決める工程。ユーザー(利用者)視点でのゴールとスコープが対象で、技術的な実現方法にはまだ踏み込まない。

- 成果物: [`REQUIREMENTS.md`](../01requirements/REQUIREMENTS.md)

## 2. 基本設計 (Basic Design / 外部設計)

「要件をどう実現するか」を、利用者から見える範囲(外部仕様)で決める工程。実装の詳細(関数名や処理の中身)にはまだ踏み込まない。

主な検討項目:

- **画面設計**: どんな画面があるか、画面同士の遷移(画面遷移図)、各画面のワイヤーフレーム
- **データ設計**: 扱うデータの構造(Atlasの場合は `index.json` のスキーマ、`/data/files/` の構成など)
- **機能一覧**: MVPで実装する機能を洗い出し、優先順位をつける
- **外部インタフェース設計**: アップロードAPIなど、フロントエンドとサーバー間でやり取りするAPIの入出力(エンドポイント、リクエスト/レスポンス形式)
- **非機能要件の具体化**: パフォーマンス、セキュリティ(iframeのsandbox方針など)、対応環境

- 成果物イメージ: `BASIC_DESIGN.md`(画面一覧・画面遷移図・API一覧・データスキーマ)

## 3. 詳細設計 (Detailed Design / 内部設計)

基本設計で決めた各機能を、実装できるレベルまで具体化する工程。開発者(実装する側)視点の資料。

主な検討項目:

- 各画面・各APIの内部処理フロー(擬似コードやシーケンス図)
- コンポーネント構成、モジュール分割の方針
- 主要な関数/クラスの入出力、責務
- バリデーション・エラーハンドリングの方針
- 例外的なケース(同名ファイルのアップロード時の挙動など、要件定義で「未決の論点」としていたものはここで確定させる)

- 成果物イメージ: `DETAILED_DESIGN.md`、または機能ごとに分割したファイル

## 4. 実装 (Implementation)

詳細設計をもとにコーディングする工程。

## 5. テスト (Testing)

- 単体テスト: 個々の関数・コンポーネント単位
- 結合テスト: 機能同士を組み合わせた動作確認(例: アップロード→タグ付け→検索が一連で動くか)
- 受け入れテスト: 要件定義の内容を満たしているかの最終確認

## 6. リリース / デプロイ (Release)

ローカルで動かす場合は「動作確認して完了」だが、将来クラウドにデプロイする場合はここでインフラ構築・デプロイ手順の整備が必要になる。

## 7. 運用・保守 (Operation & Maintenance)

リリース後の不具合対応、機能追加、依存パッケージの更新など。個人開発の場合はこの工程がそのまま次の要件定義(機能追加)に戻っていくことが多い。

---

## Atlasプロジェクトでの進め方について

上記は一般的な(比較的しっかりめの)流れだが、Atlasは個人開発・一人で実装する前提のため、そのまま全部を律儀にやると重すぎる可能性がある。実務的には以下のような簡略化がおすすめ:

- **基本設計と詳細設計を1つのドキュメントにまとめる**(`DESIGN.md`など)。画面・データ・API・処理フローをまとめて1ファイルで管理し、実装しながら随時更新する
- テストは自動テストを最小限にし、`/verify` 的な「実際に動かして確認する」運用でも十分な場面が多い
- 各工程を厳密に区切らず、「設計→実装→動作確認」を機能単位で小さく回す(アップロード機能だけ設計→実装→確認、次にタグ機能…という進め方)

次のステップとしては、`REQUIREMENTS.md` の内容をベースに、画面一覧・データスキーマ・API一覧をまとめた基本設計(簡略版の `DESIGN.md`)を作るのがよい。
