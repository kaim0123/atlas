# 開発技術 ディレクトリ構成

**位置づけ**: e-Words「テクノロジ系 開発技術」（システム開発技術・ソフトウェア開発管理技術）を、初心者が段階的に学べる解説ページ群として再構成する計画。[`03_技術要素/ディレクトリ.md`](../03_技術要素/ディレクトリ.md) の直後、ネットワーク・データベースなどの技術要素を理解したあとに「システムをどう作るか」を学ぶ想定。

**ステータス**: 移行手順ステップ1〜8を実施済み。docs教材は `01_overview`〜`04_testing` に分割し原本を `00_分割元` にアーカイブ。`src/app/sdlc/` セクションを新設（索引＋20ページ）。`test/review` を `sdlc/review` へ移動（旧URLは `RedirectTo` スタブで転送）、`sdlc/testing/techniques` を新設。`nav.ts` に `sdlc` を `dev` の直前へ挿入。既存 `test/`・`design/`・`dev/`・`ops/` から `sdlc/` への相互リンクを追加。用語集2本は計画専用のまま（ページ化せず）。

**本ファイルの構成**: 前半 §1〜10 … フォルダ内の用語集・教材から導いた**論理構成**（初学者向け20ページ案）。後半 §統合版 … `src/app` 既存ページとの**対応・ギャップ・移行手順**。

**大方針**:
- **用語集2本は計画専用**。e-Words の棚卸し・漏れ確認に使うだけで、解説ページとしては作らない。解説は教材ドラフト → ページのみ。
- 用語集の全範囲を一度にカバーしない。**全体像 → プロセス → 要件 → 設計 → 実装 → テスト → 導入・保守 → 管理**の順で、試験・実務の両方で頻出の土台を厚くする。
- 1ページは「1つの疑問に答える」粒度を目安にする。`01教材_開発プロセス.md` はライフサイクル・要件・設計・テストが同居しているため、**実装時に必ず分割**する。
- 他セクションと重なる話題は**書き分け＋相互リンク**のみ（統合しない）。本計画は**このフォルダ内の教材・用語集の整理**に留める。

**凡例**: `✓` 教材ドラフトあり / `＋` 新規執筆 / `……` はページ内トピック / `△` 用語集にあるが本フォルダでは概要または省略 / `※app` は `src/app` に既存ページあり（移動・分割・委譲の対象）/ `維持` は URL・本文を大きく変えない / 用語集は計画専用（ページ化しない）

## 全体マップ（学習順）

e-Words の2区分（システム開発技術・ソフトウェア開発管理技術）を、読者の理解の流れに並べ替えたもの。

```
Atlas
■ フェーズ3 — 開発技術（システムをどう作り、どう管理するか）
└── sdlc/    システム開発ライフサイクル
    ├── overview/        開発の全体像（工程と成果物）
    ├── process/         開発プロセスと手法（ウォーターフォール・アジャイル等）
    ├── requirements/    要件定義
    ├── design/          設計（構造化・データ中心・OO・モジュール・アーキテクチャ）
    ├── implementation/  実装
    ├── testing/         テスト（段階と技法）
    ├── review/          レビューと品質確認
    ├── deployment/      導入と受入れ
    ├── maintenance/     保守
    └── management/      開発管理（構成・変更・知的財産）

※ database/（ER・正規化）・design/（OO実装・パターン）・test/（実務テスト）・dev/（コーディング環境）を読んだあと、または並行で入る想定。
```

**想定 URL プレフィックス**: `/sdlc/...`（`src/app/sdlc/`）

**読了の目安**: `overview`〜`testing` を終えれば「開発工程と各段階の役割」、`management` を終えれば「プロジェクト管理と構成管理の基本用語」が説明できる。

# §統合版 — 既存 app との対応と移行手順

`src/app` の既存ページと本計画の `sdlc/` ツリーは、**同じ話題でも文体・目的が異なる**ものが混在している（例: `test/review` は試験用のインスペクション・ウォークスルー、`test/unit` は Vitest の書き方）。以下は「何を・どこへ・どの順で」動かすかの手順書。

**移行の大方針**:
- **上書き禁止**: 既存 app 本文は、試験用語の追記・相互リンク追加以外では削らない。
- **二層を維持**: `design/`・`test/`・`dev/`・`ops/` は「実務・コードを書くときの追補」として残す。e-Words 開発技術の試験向け土台は新設の **`sdlc/` トップセクション**に置く。
- **統合しない**: `design/architecture/*`・`dev/database/design`・`test/*` とは境界を守り、相互リンクのみ。ページの物理統合は最小限（`test/review` の移動のみ）。
- **リダイレクト**: パス移動後は `next.config` の `redirects` で旧 URL を301転送する。

**凡例（移行表）**: `分割` `移動` `改名` `統合` `新規` `維持` `アーカイブ` `逆輸出`（app → docs）`委譲`（本文は移さずリンクのみ）`短縮`（重複箇所をリンクに）

`internet` の直後・`dev` の直前に `sdlc` を挿入する（DB・ネットワークを理解したあと、コードを書く前に開発工程を学ぶ）。

```
Atlas
■ フェーズ0 — theory/
■ フェーズ1 — computer/ → network/ → internet/
■ フェーズ2 — sdlc/          ★新設（開発技術・試験主経路）
■ フェーズ3 — dev/ → design/ → test/ → security/
■ フェーズ4 — infra/ → ops/
```

**nav 挿入位置**: `src/lib/nav.ts` の `internet` と `dev` の間。
