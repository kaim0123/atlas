# 開発技術 ① 要件定義からテストまで

このノートでは、**システム開発のライフサイクル**、**要件定義と設計**、**テストの種類**を整理します。

---

## 1. 開発の流れ（ライフサイクル）

| 工程 | 主な活動 |
|------|----------|
| 要件定義 | 利害関係者から要求を聞き、**何を作るか**を文書化。機能要件・非機能要件を明確に。 |
| 設計 | 要件を実現する**仕組み**を決める。外部設計（画面・帳票）、内部設計（モジュール、DB物理設計）。 |
| 実装 | 設計に基づいてプログラムを作成。コーディング規約、単体テストをセットで。 |
| テスト | 単体→結合→システム→受入れと、範囲を広げながら検証。 |
| 導入・運用 | 本番リリース、教育、運用開始。 |
| 保守 | 是正保守（不具合修正）、適応保守（仕様変更）、予防保守。 |

**ウォーターフォール型**は「要件を固めてから設計・実装」。**アジャイル**では小さな単位で設計〜リリースを繰り返し、要求の変化に合わせて計画を見直します。

---

## 2. 要件定義で押さえること

- **機能要件**: システムが「何ができるか」（業務機能、画面、帳票）。
- **非機能要件**: 性能（応答時間、スループット）、可用性、セキュリティ、運用・保守のしやすさ。
- **ステークホルダ**: 発注者、利用部門、運用部門など。要件の合意には関係者の特定とコミュニケーションが不可欠。

---

## 3. 設計の考え方

- **モジュール化**: 結合度を低く、凝集度を高くする。
- **データ中心設計**: データ構造（ER図、正規化）を先に決め、処理を設計する。
- **オブジェクト指向設計**: クラス、継承、多態性を利用して構造化する。
- **UML**: ユースケース図、クラス図、シーケンス図、状態遷移図などで可視化。

---

## 4. テストの種類

- **単体テスト**: 個々のモジュールや関数を検証。ドライバやスタブで周辺を模擬。
- **結合テスト**: モジュール間の連携を検証。トップダウン／ボトムアップなど。
- **システムテスト**: システム全体で機能・非機能要件を検証。性能・セキュリティテストを含む。
- **受入れテスト**: 発注者・利用者が「納品してよいか」を判断するための検証。
- **回帰テスト**: 変更後に、既存機能が壊れていないかを再テスト。

要件の取り違えを防ぐことと、テストの段階的な実施が重要です。[[基本情報技術者]] の用語一覧で用語を補強し、過去問で出題のされ方に慣れていってください。
