# データベース ① 設計とトランザクション

このノートでは、**データベースの役割**、**関係モデルと3層スキーマ**、**ER図と正規化**、**トランザクションとACID**を整理します。

---

## 1. データベースの役割

- データを**一箇所で管理**し、重複や不整合を減らす。
- **複数ユーザ**が同時にアクセスしても、矛盾のない形で共有できる。
- **障害**に備えた回復機能（バックアップ、ログ、トランザクション）を提供する。
- **検索・集計**をSQLで効率よく行える。

ファイルに書き込むだけでは同時アクセス制御や整合性の保証が難しいため、**DBMS**を介してデータを扱います。

---

## 2. 関係データベースと3層スキーマ

### 2.1 関係モデル

- データは**表（テーブル）**の形で表現。1行が**レコード（タプル）**、1列が**属性（カラム）**。
- 表どうしは**主キー**と**外部キー**で関連づける。
- 操作は**関係演算**（選択・射影・結合）や**SQL**で記述。

### 2.2 3層スキーマ

| スキーマ | 役割 |
|----------|------|
| 外部スキーマ | 各アプリから見たデータの形（ビューなど） |
| 概念スキーマ | 組織全体で共通の論理構造（表の定義、関連） |
| 内部スキーマ | 物理的な格納形式（ファイル、インデックス） |

---

## 3. ER図と正規化

### 3.1 ER図

- **エンティティ**: 管理対象（顧客、商品、注文など）。
- **属性**: エンティティの性質。
- **リレーションシップ**: エンティティ間の関連（1対1、1対多、多対多）。

### 3.2 正規化

- **第1正規化**: 1セルに1つの値。繰り返し項目を別行に。
- **第2正規化**: 主キーの一部にだけ従属する属性を別表に（部分関数従属の解消）。
- **第3正規化**: 主キー以外の属性に従属する属性を別表に（推移関数従属の解消）。

正規化を進めると表が細かくなり結合が増えるため、検索性能とのバランスで非正規化することもあります。

---

## 4. トランザクションとACID

**トランザクション**は「まとめて成功するか、まとめて取り消すか」の単位です。

### 4.1 ACID特性

| 頭文字 | 意味 |
|--------|------|
| **A**tomicity（原子性） | 全体が実行されるか、全体が取り消されるか |
| **C**onsistency（一貫性） | 制約を満たした状態が保たれる |
| **I**solation（独立性） | 並行トランザクションが互いに不適切に影響しない |
| **D**urability（永続性） | コミットした結果は障害後も失われない |

### 4.2 排他制御と障害回復

- **ロック**: 更新前にロックをかけ、専有ロック・共有ロックで整合性を守る。**デッドロック**に注意。
- **ログ**: 更新前後の値や操作を記録。**ロールバック**（取り消し）や**ロールフォワード**（再実行）に使う。
- **バックアップ**: フル、差分、増分の使い分けが重要。
