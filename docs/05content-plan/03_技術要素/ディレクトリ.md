# 技術要素 ディレクトリ構成

**位置づけ**: e-Words「テクノロジ系 技術要素」（ネットワーク・データベース・セキュリティ・ユーザーインタフェース・情報メディア）を、初心者が段階的に学べる解説ページ群として再構成する計画。[`02_コンピュータシステム/ディレクトリ.md`](../02_コンピュータシステム/ディレクトリ.md) の直後、コンピュータの仕組みを理解したあとに入る想定。

**ステータス**: **ステップ1〜4完了**。ステップ1で docs 教材を `01_ネットワーク/`3本・`02_データベース/`4本に分割（原本は `00_分割元_*.md`）。ステップ2〜3で network セクションを app 上で再編（`basics`・`layers`・`topology`・`ip`・`transport`・`link`(+`wireless`)・`applications`。旧 `protocols`・`port`・`cabling`・`wifi`・`devices` はリダイレクトスタブ化）。ステップ4で **`database/` トップセクションを新設**（`basics`・`model`・`design`(dev から移動)・`sql`・`transaction`・`advanced/index`。索引ページ・`nav.ts` に新セクション追加。`dev/database` は開発者向けハブに縮小し、`physical`・`history` を残置。`/dev/database/design`→`/database/design` をスタブでリダイレクト。他セクションからの相互リンクも更新。`npx next build` 通過）。**ステップ5はフェーズA完了**（`security/basics`(CIA・脅威・マルウェア＋多層防御表を移設)・`attacks`(攻撃手法概観)・`management`(リスクマネジメント)・`countermeasures`(対策概観) を新設。`security/page.tsx` から多層防御節を basics へ移して索引をグループ順に整理。`nav.ts` の security ツリーをグループ構造に差し替え。既存16ページは**フラットのまま維持**。`npx next build` 通過）。フェーズB（`attacks/`・`auth/`・`countermeasures/` への物理移動＋旧URLリダイレクト）は**後回し**。

**ステップ6は `ui/` 完了**（`ui/` トップセクションを新設：索引＋`basics`・`gui`・`design`・`web`・`hcd` の5ページ。`nav.ts` に `ui` セクション追加(`AppWindow` アイコン、security と infra の間)。用語集§1〜§4 を元に新規執筆。`npx next build` 通過）。**`media/` は未着手**（次はここ）。用語集5本（**計画専用・ページ化しない**）は執筆済み。

> **静的エクスポートの注意**: 本プロジェクトは `output: "export"` のため `next.config` の `redirects` は使えない。旧URLは `src/components/redirect.tsx` の `RedirectTo`（クライアントリダイレクト）を置くスタブページで代替する。計画中の「`next.config` にリダイレクト追加」は、このスタブ方式に読み替える。

**大方針**:
- **用語集5本は計画専用**。e-Words の棚卸し・漏れ確認に使うだけで、解説ページとしては作らない。解説は教材ドラフト → ページのみ。
- 用語集の全範囲を一度にカバーしない。**ネットワーク → データベース → セキュリティ → UI → 情報メディア**の順で、試験・実務の両方で頻出の土台を厚くする。
- 1ページは「1つの疑問に答える」粒度を目安にする。`01教材_ネットワーク基礎.md` は階層・IP・トランスポートが同居しているため、実装時に分割する。`02教材_データベース基礎.md` も設計とトランザクションで分割を検討する。
- 他セクションと重なる話題は**書き分け＋相互リンク**のみ（統合しない）。本計画は**このフォルダ内の教材・用語集の整理**に留める。

**凡例**: `✓` 教材ドラフトあり / `＋` 新規執筆 / `……` はページ内トピック / `※app` は `src/app` に既存ページあり / 用語集は計画専用（ページ化しない）

## 既存コンテンツの移行計画（実行順）

`src/app` の既存ページと本計画のツリーは、**同じ話題でも文体・目的が異なる**ものが混在している（例: `network/protocols` は階層＋全層のプロトコル概要が同居、`dev/database` は試験用語と開発実装が同居）。以下は「何を・どこへ・どの順で」動かすかの手順書。

**移行の大方針**:
- **上書き禁止**: 既存 app 本文は、試験用語の追記・相互リンク追加以外では削らない。長いページは**分割して移す**。
- **二層を維持**: `dev/database/` は「開発者がコードを書くときの追補」として残す。e-Words 技術要素の試験向け土台は新設の **`database/` トップセクション**に置く（`dev` 配下からは切り出す）。
- **統合しない**: `internet/web`・`internet/mail`・`design/architecture/app/gui` とは境界を守り、相互リンクのみ。
- **リダイレクト**: パス改名・移動後は `next.config` の `redirects` で旧 URL を301転送する（ブックマーク・外部リンク対策）。

**凡例（移行表）**: `分割` `移動` `改名` `統合` `新規` `維持` `アーカイブ` `逆輸出`（app → docs）`委譲`（本文は移さずリンクのみ）
