# Atlas クラウド展開設計 (将来フェーズ / v0.2)

最終更新: 2026-07-03

**位置づけ**: これはMVPスコープ外の将来設計。[`REQUIREMENTS.md`](../01requirements/REQUIREMENTS.md)のMVPは引き続き**ローカル(`npm run dev`)完結**が前提であり、この文書は「将来AWSにデプロイするとしたら」の設計を先に検討しておくもの。実装着手のタイミングは別途判断する。

> v0.1からの変更点: IaCの主軸を AWS CDK 直接構築 → **AWS Blocks** 採用に変更。ただしBlocksを使う場合でもコストを最小化する構成を目指す方針を追加。

## 0. 前提

- ホスティング: AWS
- **フレームワーク: AWS Blocks を採用する**(CDKはBlocksが内包する「いつでも降りられる」エスケープハッチとして、コスト調整等に使う)
- データ永続化: 可能な限りS3(JSON)。厳しければDynamoDB、またはBlocksのDB機能を検討
- コスト: 個人利用規模で極力安く。Blocksを使う場合も**必要な機能だけを使う最小構成**にする
- アクセス制御: 現状は自分一人だが、将来的に特定の相手をメールアドレスで招待し、パスワードでログインさせて一部ページを共有する可能性がある。ページ単位で公開/非公開を分けたい

## 1. AWS Blocksについて(2026年6月プレビュー)

TypeScript製のOSSフレームワークで、DB・認証・ファイルアップロード・バックグラウンドジョブ・AIエージェント等の機能をコード記述だけで扱える。ローカルはPostgres/認証/リアルタイム機能込みの環境が自動で立ち上がり、AWSアカウント無しでも開発できる。デプロイ時はコード変更なしで本番のAWSサービス上で動作し、必要ならいつでもAWS CDKに降りて直接リソースを制御できる。Next.js含むSSRフレームワークに対応。フレームワーク自体は追加課金なし(使ったAWSサービス分のみ課金)。

本番でどのAWSサービスに対応付けられるか(DBがAurora Serverless v2かRDSか等)の詳細は、現時点の公開情報からは確認できていない。**プレビュー段階のため、実装着手時に実際に生成されるスタック構成をCDK差分等で確認し、想定通りの最小構成になっているか検証する工程を必ず挟む**(§4)。

## 2. 全体アーキテクチャ(方針)

```
┌────────────┐   HTTPS   ┌─────────────────────────┐
│  ブラウザ    │──────────▶│  AWS Blocks が構成する      │
└────────────┘            │  CloudFront + Lambda(Next.js SSR/API) │
                           └────────────┬─────────────┘
                                       │ AWS Blocksの認証機能(Cognito想定)
                                       │
                     ┌─────────────────┴─────────────────┐
                     ▼                                     ▼
        ┌─────────────────────────┐          ┌─────────────────────────┐
        │ AWS Blocksの file upload │          │ メタデータ保存             │
        │ 機能(S3想定)             │          │ 方針A: S3のJSON(推奨)     │
        │ files/<uuid>.md|html     │          │ 方針B: Blocksの database  │
        └─────────────────────────┘          │      tables機能(Postgres)│
                                              └─────────────────────────┘
```

Blocksが自動生成する内部構成はブラックボックス寄りだが、コスト最小化のために**「Blocksのどの機能を使うか」を意図的に絞り込む**ことが最大の制御ポイントになる(使っていない機能のリソースは基本的に生成されないため)。

## 3. コスト最小化の方針

### 3.1 使う機能を絞る

AtlasがBlocksの提供機能のうち実際に必要なのは以下のみ。

| Blocksの機能 | Atlasでの用途 | 使うか |
|---|---|---|
| ホスティング(Next.js SSR) | Webアプリ本体 | 使う |
| ファイルアップロード | .md/.htmlファイルの実体保存 | 使う |
| 認証 | 自分専用ログイン、将来の招待共有 | 使う |
| database tables(Postgres) | ファイル一覧・タグのメタデータ保存 | **§3.2で検討、既定では使わない** |
| バックグラウンドジョブ | 未使用 | 使わない |
| リアルタイムメッセージング | 未使用 | 使わない |
| AIエージェント | 未使用 | 使わない |

「使わない」機能はコードで参照しなければBlocksが該当リソースを生成しない想定のため、明示的に触らないことが最初のコスト対策になる。

### 3.2 メタデータ保存: DBを使わずS3のJSONで済ませる

AtlasのメタデータはローカルMVPと同じく「ファイル一覧+タグ配列」という単純なリスト構造であり、リレーショナルDB(Blocksのdatabase tables機能、おそらくPostgres系)を必要とするデータモデルではない。Postgres系リソースは待機中でも一定コストが発生しやすく、「極力安く」という方針との相性がよくない。

→ **既定方針: Blocksのdatabase tables機能は使わず、ローカルMVPと同じ `index.json` 相当のファイルをBlocksのファイルアップロード機能(S3想定)にアプリ側で書き込み/読み込みする**。これによりBlocksを使いながらもDBコストをゼロに近づける。

移行トリガー(database tables機能、またはDynamoDBへの切り替えを検討するタイミング)は以下のいずれか:

- 招待共有機能で `visibility` によるフィルタ検索が頻発し、JSON全件走査が非効率になった
- 自分以外の書き込みアクターが増え、S3の条件付きPutObject(`If-Match`)による競合検知だけでは不安になった
- ファイル数が数千件規模になった

### 3.3 それでもBlocksの内部がDBを前提にしていた場合

Blocksのホスティング/認証/ファイルアップロード機能自体が内部的にPostgresを共有前提で構成する可能性もゼロではない。その場合の対応方針:

1. Blocksが「いつでもCDKに降りられる」設計であることを利用し、生成されたCDKスタックを確認する
2. DB関連リソースが見つかった場合、最小容量設定(最小ACU、オートポーズ/スケールtoゼロ等)に調整できないか確認する
3. 調整が難しい、またはコストが許容範囲を超える場合は、Blocksの当該機能(database tables)を使わない構成に倒し、§3.2のS3 JSON方式で運用する

## 5. 認証・認可

- Blocksの認証機能を使う(内部的にCognito相当と想定)
- セルフサインアップは無効化し、招待制で運用する(自分がアカウントを作成し、招待したい相手にメールで通知)
- ロールは `owner`(自分、全件アクセス可)と `viewer`(招待された相手、`visibility: shared` のページのみ閲覧可)の2段階
- 認可チェックはNext.js側のmiddlewareで行い、Blocksが発行するセッション/トークンを検証する

## 6. セキュリティ設計

Blocksを使う場合でも、以下の方針はv0.1から変更しない。

- **アップロード**: 拡張子は `.md` / `.html` のみ許可。Markdownは `rehype-sanitize`、HTMLは `iframe sandbox`(`allow-scripts`なし)で表示する
- **公開範囲**: S3相当のストレージはBlocksの管理下に置き、直接の公開URLを露出させない(Blocksのホスティング機能経由のみでアクセスさせる)
- **シークレット**: Blocksが管理するシークレット機構(環境変数/Parameter Store相当)を使い、コードにハードコードしない
- **監視・コストガードレール**: AWS Budgetsで低めの閾値(例: 月$5)にアラームを設定し、想定外の課金増を早期検知する。BlocksがCDKベースであることを踏まえ、生成されたリソースにもBudgetsのアラート対象を確実に含める
- **AWS WAF**: 個人規模・招待制ログイン前提のため初期導入は見送り、不審なアクセスが増えた場合に追加を検討する

## 7. データモデルの拡張(共有機能向け)

将来の招待共有に備え、メタデータ(`index.json` またはdatabase tables採用時のテーブル)に `visibility: "private" | "shared"` を追加する。`owner`は全件閲覧可、`viewer`は `shared` のみ閲覧可という単純な2段階から始める。

## 8. コスト試算(目安・個人利用+数名の招待、低トラフィック想定)

Blocksの内部構成が現時点で完全には分からないため幅を持たせた目安。

| 項目 | 目安 |
|------|------|
| ホスティング(Lambda+CloudFront相当) | 無料枠内が中心でほぼ$0(個人利用の低トラフィック) |
| ファイルストレージ(S3相当、数百件・数GB未満) | 数十円/月 |
| 認証(Cognito相当、無料枠50,000 MAU) | $0 |
| メタデータ(§3.2の方針でDB不使用の場合) | $0(S3の範囲内) |
| メタデータ(database tables機能を使う場合) | **要検証。待機コストが発生する可能性があり、§3.3の調整を検討** |
| Route 53(独自ドメインを使う場合) | $0.5/月 + ドメイン登録費用 |
| **合計目安(§3.2の方針採用時)** | **月0〜数百円程度** |

## 9. 段階移行プラン

1. **フェーズ0(現在)**: ローカル完結(`npm run dev`)、DBなし、認証なし([`REQUIREMENTS.md`](../01requirements/REQUIREMENTS.md)のMVP)
2. **フェーズ1(クラウド・自分専用)**: AWS Blocksでホスティング+認証+ファイルアップロードのみ導入(`owner`ロールのみ)、メタデータはS3 JSON方式(§3.2)
3. **フェーズ2(招待共有)**: `visibility` フィールド追加、`viewer`ロール追加、招待フロー実装。必要であればdatabase tables機能やDynamoDBへの移行を検討

## 10. 未決の論点

- Blocksの本番構成が実際にどのAWSサービスに対応するか(§4の検証タスクで確認する)
- メタデータのS3 JSON書き込みをBlocksのファイルアップロード機能経由で行う場合の、アプリ側からの読み書きAPIの扱い(Blocksの標準機能で足りるか、CDKでカスタムLambdaを追加する必要があるか)
- 招待した相手ごとのページ単位ACL(全体共有だけで足りるか、個別指定が必要になるか)は利用実態を見てから判断
- カスタムドメインを使うかどうか
