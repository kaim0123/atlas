# Atlas 基本設計 (v0.1)

最終更新: 2026-07-03

> **注記(2026-07-10)**: 本書が前提とするアップロード・タグ編集・ファイル詳細画面などはAtlas本体には実装しない方針に変更した。現在のAtlasの実際の画面構成・コンポーネント設計は [`03detail-design/site-detail.md`](../03detail-design/site-detail.md) を参照。本書は将来アップロード機能を別アプリとして切り出す場合の参考資料として残している。

前提: [`REQUIREMENTS.md`](../01requirements/REQUIREMENTS.md) のMVPスコープに基づく。ここでは「利用者から見える仕様」を定義し、実装の中身(関数構成など)には踏み込まない(詳細設計で扱う)。

## 0. 本設計での前提確認事項

要件定義以降、以下を決定した。

- **タグ付けのタイミング**: アップロード時に指定できる(複数ファイル共通タグ)、かつアップロード後もファイル詳細画面から個別に編集できる、両対応
- **タイトルの取得元**: ファイル名(拡張子除く)をそのまま使う。自動抽出(`<title>`や`# 見出し`)は行わない。この方針により、ウィキリンク `[[ページ名]]` の解決もファイル名(拡張子除く)との一致で行う
- **アップロード単位**: 複数ファイルの一括アップロードに対応する
- **ホーム画面**: タグツリーナビゲーションを中心に据える

## 1. 画面一覧

| # | 画面名 | パス(想定) | 概要 |
|---|--------|-----------|------|
| 1 | ホーム(タグブラウズ) | `/` | サイドバーのタグツリーで絞り込みつつファイル一覧を見る、初期画面 |
| 2 | ファイル詳細 | `/files/[id]` | md描画 or html iframe表示。タグ編集・裏リンク一覧を表示 |
| 3 | アップロード | モーダル(全画面共通で起動可能) | 複数ファイル選択+共通タグ指定→アップロード実行 |
| 4 | 検索結果 | `/search?q=...` | ヘッダー検索ボックスからの遷移先。タイトル/タグ/本文でのマッチ結果一覧 |

共通レイアウト(ホーム・詳細・検索結果の3画面で共通):

```
+----------------------------------------------------+
| Header: [Atlasロゴ] [検索ボックス]      [アップロード]|
+------------------+-----------------------------------+
| Sidebar          | Main                              |
| (タグツリー)       | (画面ごとに切り替わる)              |
|                   |                                    |
+------------------+-----------------------------------+
```

- Header・Sidebarは全画面で常時表示(htmlファイル閲覧時もiframeはMain領域内に収まる)
- Sidebarのタグツリーは「すべて(未分類含む)」をルートに持つ

## 2. 画面遷移図

```
[ホーム "/"]
  ├─ サイドバーのタグをクリック ─→ [ホーム "/"] (選択タグでMainを絞り込み、同一画面内で状態遷移)
  ├─ ファイル一覧のアイテムをクリック ─→ [ファイル詳細 "/files/[id]"]
  ├─ ヘッダーの検索ボックスに入力してEnter ─→ [検索結果 "/search?q=..."]
  └─ ヘッダーの「アップロード」ボタン ─→ [アップロードモーダル]
                                              └─ アップロード完了 ─→ [ホーム "/"] (アップロードしたファイルをハイライト)

[ファイル詳細 "/files/[id]"]
  ├─ 本文中のウィキリンク/内部リンクをクリック ─→ [ファイル詳細 "/files/[別id]"]
  ├─ 裏リンク一覧のアイテムをクリック ─→ [ファイル詳細 "/files/[別id]"]
  ├─ サイドバーのタグ/検索/アップロード ─→ (ホームと同じ遷移)
  └─ タグ編集欄でタグを追加/削除 ─→ (同一画面内、保存後にサイドバーのタグツリーにも反映)

[検索結果 "/search?q=..."]
  └─ 結果アイテムをクリック ─→ [ファイル詳細 "/files/[id]"]
```

## 3. 機能一覧(優先度)

| 機能 | 優先度 | 備考 |
|------|--------|------|
| 複数ファイル一括アップロード(.md/.html) | 必須 | ドラッグ&ドロップ + ファイル選択ダイアログ |
| アップロード時の共通タグ指定 | 必須 | 空欄可(未分類として保存) |
| ファイル詳細画面でのタグ編集 | 必須 | 追加・削除。既存タグ候補のオートコンプリート |
| タグツリーナビゲーション(サイドバー) | 必須 | `親/子/孫` を階層表示、各ノードに件数表示 |
| ファイル一覧表示(タグ絞り込み対応) | 必須 | タイトル・タグ・アップロード日時を表示 |
| Markdown描画 | 必須 | remark/rehypeでレンダリング |
| ウィキリンク `[[ページ名]]` 解決 | 必須 | ファイル名(拡張子除く)との一致で解決。未一致はリンク切れ表示 |
| HTML表示(iframe sandbox) | 必須 | Header/Sidebar維持、Main領域に埋め込み |
| 裏リンク表示 | 必須 | mdはウィキリンク解析、htmlは`<a href>`のファイル名一致で計算 |
| 検索(タイトル/タグ/本文) | 必須 | 実装方式(都度走査 or 簡易インデックス)は詳細設計で決定 |
| ファイルの削除・上書き等のCRUD | 対象外 | 要件定義どおりMVPでは行わない |

## 4. データ設計

### 4.1 ディレクトリ構成

```
/data
  /files/                 -- アップロードされた実体ファイル(フラット、サブフォルダなし)
    <uuid>.md
    <uuid>.html
  index.json              -- 全ファイルのメタデータ一覧
```

- 実体ファイルの物理ファイル名は `id` (UUID) + 元拡張子とし、表示用タイトルは `index.json` の `filename`(アップロード時の元ファイル名)を使う。これにより同名ファイルを複数回アップロードしても物理ファイル同士は衝突しない
  - 例外挙動(同名ファイル再アップロード時にどう見せるか)は運用上「別ファイルとして共存」を基本方針とし、詳細設計でUI上の注意表示を検討する

### 4.2 `index.json` スキーマ

```json
{
  "files": [
    {
      "id": "uuid",
      "filename": "元のファイル名(拡張子含む、表示タイトルはここから拡張子を除いたもの)",
      "storedAs": "uuid.md | uuid.html",
      "format": "md | html",
      "tags": ["親/子/孫", "..."],
      "uploadedAt": "ISO8601"
    }
  ]
}
```

- `tags` は空配列可(未分類)
- タイトルは保存せず、`filename` から都度導出する(方針0参照)

### 4.3 タグツリーの導出

- `index.json` の全ファイルの `tags` を集計し、`/` 区切りでパースしてツリー構造をメモリ上に構築する(永続化はしない、常に `index.json` から再生成可能)
- 同じタグパスの重複や大文字小文字の揺れは正規化(小文字化などは詳細設計で決定)

## 5. 外部インタフェース設計(API)

MVPでは以下のRoute Handlerを想定。

| メソッド | パス | 概要 | リクエスト | レスポンス |
|---------|------|------|-----------|-----------|
| POST | `/api/files` | 複数ファイルアップロード | `multipart/form-data`: `files[]`, `tags`(共通タグ、カンマ区切り文字列 or 配列) | 作成された各ファイルのメタデータ配列 |
| GET | `/api/files` | ファイル一覧取得 | クエリ: `tag`(絞り込み), `q`(検索語、任意) | ファイルメタデータ配列 |
| GET | `/api/files/:id` | 単一ファイルのメタデータ+本文取得 | - | メタデータ + 本文(text) |
| PATCH | `/api/files/:id/tags` | タグ更新 | `{ "tags": string[] }` | 更新後のメタデータ |
| GET | `/api/files/:id/backlinks` | 裏リンク一覧取得 | - | 参照元ファイルのメタデータ配列 |
| GET | `/api/tags` | タグツリー取得 | - | 階層構造 + 各ノードのファイル件数 |
| GET | `/api/search` | 検索 | クエリ: `q` | マッチしたファイルメタデータ配列 |

- `GET /api/files/:id` はhtml表示のiframe `srcdoc` にそのまま本文を渡す用途にも使う。将来的に大きいHTMLファイルが増えた場合は `srcdoc` ではなく専用の静的配信エンドポイント(例: `/api/files/:id/raw`)への切り替えを検討する

## 6. 非機能要件の具体化

- **対象環境**: ローカル実行(`next dev` / `next start`)。将来クラウドデプロイする場合はファイルシステムへの書き込み前提が崩れるため、その際にストレージ方式を再設計する
- **想定データ規模**: 個人利用のため数百ファイル程度を想定。`index.json` の全件走査や都度検索でも十分な性能が出る規模
- **セキュリティ**:
  - HTMLは`iframe sandbox`(`allow-scripts`なし)で隔離表示し、スクリプト実行を許可しない
  - Markdownは`rehype-sanitize`でHTML注入を防ぐ
  - アップロードは拡張子を`.md`/`.html`のみ許可し、それ以外は拒否する
- **対応ブラウザ**: 最新版のモダンブラウザ(Chrome/Safari/Edge)のみ考慮。個人利用のため後方互換は不要

## 7. 未決の論点(詳細設計で決定)

- 検索の実装方式(都度スキャン vs 簡易転置インデックス)、日本語の部分一致・形態素解析の要否
- タグ名の正規化ルール(大文字小文字、全角半角スペースの扱いなど)
- 同名ファイル再アップロード時のUI上の見せ方(別ファイルとして両方一覧に出す際の視認性)
- ウィキリンクが複数ファイルに一致した場合(同名ファイルが複数存在する場合)の解決優先順位
