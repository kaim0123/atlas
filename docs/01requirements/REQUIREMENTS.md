# Atlas 要件定義 (v0.1 / MVP)

最終更新: 2026-07-03

> **注記(2026-07-10)**: 本書が前提とする「アップロード・タグ管理・裏リンク」機能はAtlas本体には実装しない方針に変更した。現在のAtlasはコンテンツを `page.tsx` として直接コード化する**静的サイト**であり、経緯は [`00devflow/STATIC_DEPLOY_PLAN.md`](../00devflow/STATIC_DEPLOY_PLAN.md) を参照。本書はアップロード機能を将来別アプリとして切り出す場合の参考資料として残している。

## 目的

個人の知識地図・Wikiとして使うWebアプリ。ローカルで動く「閲覧・検索・タグ管理」中心のツールで、編集自体は外部エディタ(VSCodeなど)で行う前提。

実際に投入する最初のコンテンツ(コンピュータ・ソフトウェア開発の解説サイト、全23ページ)の構成は[`05content-plan/CONTENT_STRUCTURE.md`](../05content-plan/CONTENT_STRUCTURE.md)を参照。

## MVPでやること

- `.md` / `.html` ファイルのアップロード
- アップロードしたファイルへのタグ付け(DB不要、JSONで管理)
- タグを `親/子/孫` のようなパス表記で階層化し、疑似フォルダとして閲覧できるようにする
- ウィキリンク `[[ページ名]]` と裏リンク(backlinks)の表示(mdファイル対象)
- 全文/タグ検索
- 閲覧専用。CRUD・リッチテキストエディタは実装しない

## MVPでやらないこと(将来検討)

- データベース接続(将来的にSQLite等へ移行する可能性はあるが、初期は不要)
- `.html` ファイル内での `[[]]` ウィキリンク記法対応(スキップ。通常の `<a href>` を使い、ファイル名一致で裏リンクを計算する)
- クラウドデプロイ(まずはローカルで動かす。将来的にデプロイする可能性あり。設計は [`CLOUD_DEPLOYMENT_DESIGN.md`](../04CICD/CLOUD_DEPLOYMENT_DESIGN.md) を参照)

## データ保存

保存場所はプロジェクト内の `/data` (Gitでの管理・バックアップがしやすいことを優先)。

- `/data/files/` — アップロードされた `.md` / `.html` の実体。サブフォルダを作らずフラットに保存する
- `/data/index.json` — ファイルごとのメタデータ一覧

```json
{
  "id": "string (uuid等)",
  "filename": "string",
  "format": "md | html",
  "title": "string",
  "tags": ["親/子/孫", "..."],
  "uploadedAt": "ISO8601"
}
```

- `tags` は `親/子` 形式のパス文字列。UI側でパースしてツリー化し、疑似フォルダとして表示する
- 裏リンクは `index.json` には保存せず、閲覧時にファイル内容をスキャンして動的計算する(将来的にキャッシュ化してもよいが、常に再生成可能な状態を保つ)

## 表示ルール

### Markdown

- `remark`/`rehype` パイプラインでレンダリングする
- `[[ページ名]]` をパースし、`index.json` の `title`/`filename` と突き合わせて内部リンクに変換する
- あるページを開いた際、そのページを参照している他ページの一覧(裏リンク)をサイドパネル等に表示する

### HTML

- Header/Sidebarは維持したまま、Main領域に `<iframe sandbox srcdoc="...">` で埋め込み表示する
- 採用理由: 独自の `<head>`/`<style>` を持つ完全なHTMLドキュメントを、Atlas自身のCSSと衝突させずに安全に表示できるため

**HTMLファイル執筆時のルール**

1. 単体で完結させる。外部CDNへの依存やscriptタグは極力避ける。画像はBase64埋め込みを推奨(相対パスの画像・リンクはAtlas上では解決されない)
2. 内部リンクは通常の `<a href="他のファイル名.html">` でよい。ファイル名の一致で裏リンクを計算する
3. デフォルトではJSは実行不可(`sandbox` 属性に `allow-scripts` を付けない方針)。動くデモ等でJS実行が必要な場合は個別に相談・検討する
4. `target="_blank"` の外部リンクは通常通り機能する。iframe内から親ウィンドウ(サイドバー等)を操作することはできない

## 技術スタック

- Next.js (App Router) + React + TypeScript
- Tailwind CSS
- Markdown処理: `remark` / `rehype` (+ `rehype-sanitize`)
- HTML表示: `iframe sandbox` (DOMPurify等によるサニタイズは不要、iframeで隔離するため)
- ファイルアップロード: Next.js Route Handler + FormData
- データ永続化: `/data` 配下のファイル + `index.json`。DBは使わない
  - 将来DB化する場合はSQLiteへの移行を想定し、`index.json` のスキーマをそのままテーブル設計に転用しやすい形にしておく

## 未決の論点
